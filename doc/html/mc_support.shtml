Multi-core/multi-thread support in SLURM

(Draft Aug 17,2006)

This is a first draft of the multi-core support in SLURM. We will enhance the documentation for the final 1.2 release. This draft is so that early testers have some preliminary documentation at their disposition.

Overview of new srun flags

The new flags allow users to better take advantage of the new architecture since users are able to explicitly specify the number of sockets, cores, and threads required by their application.

It is important to note that many flags are only meaningful if the processes' affinity is set. In order for the affinity to be set the task/affinity plugin must be enabled. The affinity can be set to either a specific logical processor (socket, core, threads) or at a higher level than the lowest level of logical processor (core or thread). In the later case the processes are allowed to roam within a specific socket or core. We introduce the notion of fat masks to describe affinity masks which bind a process to more than one logical processor.

--cpu_bind

Specify type of process affinity binding if any

--sockets-per-node

Number of sockets in a node to dedicate to a job (minimum or range)

--cores-per-cpu

Number of cores in a CPU to dedicate to a job (minimum or range)

--threads-per-core

Number of threads in a core to dedicate to a job (minimum or range)

Extensions to -N

Specify -N [min-max]nodes:[min-max]sockets_per_node:[min-max]cores_per_sockets:[min-max]threads_per_cores

Extensions to --distribution (-m)

-m, --distribution=(hostfile|block|cyclic|plane=x|block:cyclic)

--mincores

Nodes must meet this minimum number of cores per socket

--minthreads

Nodes must meet this minimum number of threads per core

--cpu_bind: Explicit binding interface

The following srun flag provides a low-level core binding interface:

--cpu_bind=    Bind tasks to CPUs

q[uiet],          quietly perform binding before task runs (default)

v[erbose],       verbosely report binding before task runs

no[ne]              don't bind tasks to CPUs

thread,              bind at the thread level (currently the default. Could change!).

core,                bind at the core level

socket,             bind at the socket level

rank                 bind by task rank

map_cpu:<list> bind by mapping CPU IDs to tasks as specified

                     where <list> is <cpuid1>,<cpuid2>,...<cpuidN>

mask_cpu:<list>bind by setting CPU masks on tasks as specified

                     where <list> is <mask1>,<mask2>,...<maskN>

Examples:

srun –n 8 –N 4 –cpu_bind=mask_cpu:0x1,0x4 a.out

srun –n 8 –N 4 –cpu_bind=mask_cpu:0x3,0xD a.out

srun –n 8 –N 4:2:1:1 –m block:cyclic –cpu_bind=core   a.out (+ high Level flags)

srun –n 8 –N 4:2:1:1 –m block:cyclic –cpu_bind=socket a.out (+ high Level flags)

See also srun --help

-N [min-max]nodes:[min-max]sockets_per_node :[min-max]cores_per_sockets:[min-max]threads_per_cores

We have updated the node selection infrastructure with a mechanism that allows selection of logical processors at a finer granularity. Users are able to request a specific number of nodes, sockets,  cores, and threads:

-N N[:S[:C[:T]]]expands to:

--nodes=N             number of nodes to allocate

--sockets-per-node=S        number of sockets per node to allocate

--cores-per-socket=C        number of cores per socket to allocate

--threads-per-core=T         number of threads per core to allocate

            each field can be 'min[-max]' or wildcard '*'

             total cpus requested = (N x S x C x T)

Examples:

srun –n 1]6 –N 4 <additional args> a.out

srun –n 16 –N 4:2:2:1 <additional args> a.out

srun –n 16 –N 4:2-2:2-2:1-1 <additional args> a.out

OR

srun –n 16 –N 4 –sockets-per-node 2-2 –cores-per-socket 2-2 –threads-per-core 1-1 <additional args> a.out

srun –n 16 –N ‘2-4:1-2:*:1’ <additional args> a.out

srun –n 16 –N ‘4:2-2:*:1’ <additional args> a.out

srun –n 16 –N ‘4:4:1-1:1-1’ <additional args> a.out

Note:

  • Adding –cpu_bind=no to the command line will NOT bind the processes to the logical processes.
  • Binding is on by default when the extensions to –N are used. The default binding on multi-core/multi-threaded systems is equivalent to –cpu_bind=thread (lowest level of logical processor). This is equal to –cpu_bind=core on a single threaded system.

See also srun –help

New distributions: Extensions to –distribution

-m, --distribution=type distribution method for processes to nodes and to lowest level of logical processors (type = block|cyclic|plane=x|hostfile:block|cyclic)

The new plane distribution results in a block cyclic distribution of blocksize equal to x. In the following we use “lowest level of logical processors” to describe sockets, cores or threads depending of the architecture. The new distribution divides the cluster into planes (including a number of the lowest level of logical processors on each node) and then schedule first within each plane and then across planes.

The second distribution (after ”:”) allows users to specify a distribution method for processes within a node and applies to the lowest level of logical processors (sockets, core or thread depending on the architecture).

The binding is enabled automatically when high level flags are used as long as the task/affinity plug-in is enabled. To disable binding at the job level use –cpu_bind=no.

The distribution flags can be combined with the other switches e.g.

srun –n 16 –N ‘4:2:*:1’ -m block:cyclic –cpu_bind=socket a.out

srun –n 16 –N ‘4:2:*:1’ -m plane=2 –cpu_bind=core a.out

srun –n 16 –N ‘4:2:*:1’ -m plane=2 a.out

The default distribution on multi-core/multi-threaded systems is equivalent to -m block:cyclic with –cpu_bind=thread.

See also srun –help

Motivation behind high-level srun flags

The motivation behind allowing users to use higher level srun flags instead of --cpu_bind is that the later can be difficult to use. The proposed high-level flags are easier to use than --cpu_bind because:

  • Affinity mask generation happens automatically when using the high-level flags.
  • The length and complexity of the --cpu_bind flag vs. the length of the combination of -N and --distribution flags make the high-level flags much easier to use.

Also as illustrated in the example below it is much simpler to specify a different layout using the high-level flags since users do not have to recalculate mask or CPU IDs. The new approach is very effortless compared to rearranging the mask or map.

Given a 32-process MPI job and a four dual-CPU dual-core node cluster, we want to use a block distribution across the four nodes and then a cyclic distribution within the node across the physical processors. We have had several requests from users that they would like this distribution to be the default distribution on multi-core clusters. Below we show how to obtain the wanted layout using 1) the new high-level flags and 2) --cpubind

High-Level flags

Using SLURM's new high-level flag, users can obtain the above described layout on XC with:

# mpirun -srun -n 32 -N 4:4:2 --distribution=block:cyclic a.out

or

# mpirun -srun -n 32 -N 4:4:2 a.out

(if –distribution=block:cyclic becomes the default distribution)

Low-level flag –cpu_bind

Using SLURM's –cpu_bind flag, users must compute the CPU IDs or taskset masks as well as make sure they understand the core numbering on their system. Another problem arises when core numbering is not the same on all nodes. The ‑‑cpu_bind option only allows users to specify a single mask for all the nodes. Using SLURM high-level flags remove this limitation since SLURM will correctly generate the appropriate masks for each requested nodes.
